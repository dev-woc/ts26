# Plan: Opportunity Brief (Phase 4.1)

## Goal
Add an AI-generated "Know Before You Search" brief to the top of the Summary tab — answering what the government is buying, where, who qualifies, key deliverables, and any gotchas — before the user takes any other action.

## Success Criteria
- [ ] Brief renders at the top of the Summary panel for any opportunity
- [ ] Brief is generated by GPT-4o from `parsedAttachments` + `rawData` + `description`
- [ ] Generated brief is cached in `opportunityBrief` JSON field (no re-generate on reload)
- [ ] If no brief exists, "Generate Brief" button is shown; spinner during generation (~10–15s)
- [ ] All 4 required sections render: What They're Buying, Place of Performance, Who Qualifies, Key Deliverables + Period
- [ ] "Heads Up" section shows for: bonding/insurance, security clearance, set-aside, <30-day timeline, mandatory on-site
- [ ] No TypeScript errors (`npx tsc --noEmit`)

## Design Decisions (from PRD §3.18)
| # | Decision |
|---|----------|
| 4 | Brief at top of Summary tab; attachments + details move below |
| 5 | GPT-4o from `parsedAttachments` + `rawData`; cached to `opportunity.opportunityBrief` |
| 6 | Sections: (1) What they're buying, (2) Place of performance, (3) Who qualifies, (4) Key deliverables + period |
| 7 | Heads Up triggers: bonding/insurance >$100K, security clearance (Secret/TS/DD-254), set-aside eligibility, unusual timelines (<30 days) or mandatory on-site |

## Brief JSON Structure
```typescript
interface OpportunityBrief {
  whatTheyAreBuying: string           // Plain English, 2–4 sentences
  endUser?: string                    // Who benefits from the work
  placeOfPerformance: {
    location: string                  // "Fort Knox, KY"
    siteType: 'on-site' | 'remote' | 'hybrid' | 'unknown'
    travelRequired: boolean
  }
  whoQualifies: {
    setAside?: string                 // "SDVOSB", "8(a)", etc. or null
    licenses?: string[]               // ["KY Electrical License"]
    clearances?: string[]             // ["Secret"]
    certifications?: string[]         // ["ISO 9001"]
  }
  keyDeliverables: Array<{
    item: string
    frequency?: string                // "Monthly", "Per incident", "Annually"
  }>
  periodOfPerformance: {
    basePeriod: string                // "1 year (Oct 2026 – Sep 2027)"
    optionYears?: number
  }
  estimatedValue?: string             // "$2.4M/year (stated)" or "Not stated"
  contractType?: string               // "FFP", "T&M", etc.
  headsUp: Array<{
    type: 'bonding' | 'clearance' | 'setaside' | 'timeline' | 'onsite' | 'other'
    message: string                   // Human-readable warning
  }>
  generatedAt: string                 // ISO timestamp
}
```

## Validation Strategy

### Automated
- `npx tsc --noEmit` — must be clean

### Manual User Journey
1. Open any opportunity workspace → Summary tab is default
2. If brief not yet generated: see "Generate Brief" button at top of Summary tab
3. Click "Generate Brief" → button shows spinner, ~10–15s
4. Brief card appears with all 4 sections populated
5. Reload the page → brief loads instantly from cache (no re-generate)
6. Brief shows "Heads Up" block if security clearance, bonding, SDVOSB set-aside, or tight timeline is detected
7. Attachments and opportunity metadata still visible below the brief card

## Notes
- GPT-4o model (not mini) — brief needs to be accurate and complete, not fast
- If `parsedAttachments` is null, still generate using `rawData.description` + `opportunity.description` — do not block on parsing
- Do not auto-generate on load — user triggers with button; brief cached permanently after first generation
- "Regenerate" button (small, secondary) shown once brief exists, for refresh if data changes
- `headsUp` array may be empty — only render the Heads Up block if `headsUp.length > 0`
